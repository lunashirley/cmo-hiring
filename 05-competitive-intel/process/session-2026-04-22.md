# Build Session: Competitive Intelligence Pipeline
**Date:** 2026-04-22
**Participants:** Luna (product), Claude (engineering)

---

## What We Built

A fully automated competitive intelligence pipeline deployed to `lunashirley/cmo-hiring/05-competitive-intel`. It runs weekly on GitHub Actions, scrapes competitor sources via Firecrawl, analyzes them with Claude against a manually maintained company profile, and commits a battlecard PDF + changelog entry back to the repo.

### Final architecture

```
GitHub Actions (weekly cron + manual dispatch)
  → scrape.py: Firecrawl REST API → clean markdown per source
  → analyze.py: Claude Haiku → battlecard.md + changelog entry (XML tag output)
  → render.py: Python markdown lib → battlecard.html → WeasyPrint → battlecard.pdf
  → notify.py: Slack webhook (optional, silent skip if not configured)
  → git commit → output/ in cmo-hiring repo
```

### Key design decisions made during the session

- **Single repo** (not two-repo) — Luna's repo `cmo-hiring` already existed; business data lives alongside scripts, repo kept private
- **GitHub Actions over Claude Code Scheduled Agents** — more reliable scheduling, native git change tracking, lower cost
- **Haiku over Sonnet** — ~$1.30/year vs ~$4.70/year; quality sufficient for structured analysis with explicit prompt schema
- **XML tags over JSON** for Claude output — HTML content inside JSON causes escaping/truncation issues; XML is robust
- **Python `markdown` library over Claude for HTML** — eliminates a second API call and all associated token limits
- **Firecrawl REST API over SDK** — SDK had breaking changes across versions; REST API is stable

## Debugging Log

The pipeline required several rounds of fixes before running cleanly. Documented here as a learning record.

### Round 1 — JSON parse error
**Error:** `JSONDecodeError: Unterminated string`
**Cause:** Claude's output (markdown battlecard + HTML in a JSON object) exceeded `max_tokens=4096` and was truncated mid-string.
**Fix:** Switched from JSON to XML tag output format (`<battlecard_md>`, `<changelog_entry>`). Raised max_tokens to 8192.
**What this revealed:** Asking Claude to return large HTML inside JSON is structurally fragile. XML tags are the right format for large, mixed-content outputs.

### Round 2 — Firecrawl SDK v1 → v2 breaking change
**Error:** `'Firecrawl' object has no attribute 'scrape_url'` then `'FirecrawlApp' object has no attribute 'scrape'`
**Cause:** The `firecrawl-py` SDK changed its class name and method signature between versions. Pinning to `>=1.0.0` then `>=2.0.0` both caused issues because the installed version didn't match assumptions.
**Fix:** Dropped the SDK entirely. Switched to direct Firecrawl REST API calls via `requests`. No more SDK version dependency.
**What this revealed:** For any third-party SDK that's under active development, REST API calls are more durable than SDK wrappers.

### Round 3 — HTML generation truncated at max_tokens
**Error:** `ValueError: Could not find <battlecard_html> block` after `Warning: Call 2 hit max_tokens`
**Cause:** The second Claude call (analysis → HTML) also hit the 4096 token limit. A full styled HTML document is large.
**Fix:** Removed the second Claude call entirely. Replaced with Python `markdown` library + hardcoded CSS template in `render.py`. Reliable, fast, and free.
**What this revealed:** Using Claude to generate HTML is wasteful when a deterministic renderer produces consistent results at zero token cost.

### Round 4 — Manual content truncation was wrong
**Error:** No error — Luna flagged it.
**Cause:** After the HTML truncation fix, a `MAX_CHARS_PER_SOURCE = 6000` cap was added to limit input size. Luna correctly pointed out that Firecrawl already handles content sizing — this truncation could silently drop important competitor signals.
**Fix:** Removed manual truncation. Added `onlyMainContent: true` to the Firecrawl request, which is the purpose-built parameter for filtering nav/ads/footers.
**What this revealed:** Trust the tool you're using. Firecrawl exists precisely to return clean, sized content — working around it is the wrong layer to intervene.

### Round 5 — Claude wrapping XML output in code fences
**Error:** `ValueError: Could not find <battlecard_md> block`; raw response started with ` ```xml `
**Cause:** Even with explicit instructions not to, Claude (Haiku) occasionally wraps XML output in markdown code fences. When the response was also truncated, the closing fence never appeared, making extraction fail.
**Fix:** Added `_strip_code_fences()` function applied before every `_extract_tag()` call.

### Round 6 — Old workflow re-runs
**Pattern:** Luna triggered "Re-run" on failed GitHub Actions jobs, which replays the exact same commit SHA — not the latest code.
**Fix:** Not a code fix. Established the practice: always trigger a **new run** via "Run workflow", never re-run a failed job when a code fix has been pushed.

## What Went Well

**Architecture decisions held up.** The core choices — GitHub Actions, single repo, Firecrawl for scraping, Claude for analysis, WeasyPrint for PDF — were all solid and didn't need revisiting.

**Incremental debugging was effective.** Each error gave enough signal to identify the root cause. The raw response preview added to `_extract_tag` error messages was particularly useful for diagnosing the XML/code-fence issue.

**Luna's instincts were right.** The pushback on manual content truncation (Round 4) was correct — trusting Firecrawl's `onlyMainContent` is the right abstraction. The feedback to fix everything in one commit (not one-by-one) was also right and led to better commit hygiene in later rounds.

**Slack left as a one-secret activation.** Designing notify.py to silently skip when `SLACK_WEBHOOK_URL` is absent means it's zero-friction to enable later — no code changes needed.

---

## What Could Have Been Better

**Firecrawl SDK should have been avoided from the start.** The REST API was always the more durable choice. The SDK version ambiguity caused two separate debugging rounds. A quick check of the SDK's release history would have flagged this risk upfront.

**max_tokens should have been set to 8192 from the start.** The battlecard schema is detailed (pricing table, five narrative sections, strategic implications, objection handling table). Estimating output token requirements before setting the limit would have avoided Round 1 entirely.

**Two-call Claude approach added unnecessary complexity.** Splitting into analysis + HTML generation sounded clean but created a second truncation surface. The right answer — use a deterministic HTML renderer — was available from the start.

**Re-run vs. new run wasn't communicated early enough.** This caused Luna to hit old errors on new runs. Should have been called out the first time a fix was pushed.

**Code fence stripping should have been defensive from day one.** Claude wrapping outputs in fences despite instructions is a known model behavior. `_strip_code_fences()` should have been in the first version of `_extract_tag`.
